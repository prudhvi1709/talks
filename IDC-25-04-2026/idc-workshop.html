Idea Accelerators

Don't build a generic chatbot. Pick one messy input, one sharp user, one surprising output.

Use Your Own Data

  • Bank statement → money story
  • Calendar + deadlines → student cockpit
  • Screen time / health logs → behavior mirror

Use A Live Signal

  • Camera → posture or drowsiness coach
  • Voice / journal → dream or mood mapper
  • Location / meetings → timely assistant

Make It Weird Enough To Share

  • Text / diary → soundtrack or trailer
  • Group chat → roast or memory map
  • Unread PDF → visual explainer

Best prompts start with: "Take this input, do one smart thing, return something I can use or share."

Push them away from generic chatbots and toward specific transformations. Good examples to mention if the room is blank: bank statement to spending story, savings leaks, and monthly advice. class schedule plus deadlines into a student dashboard. screen time, Netflix, YouTube, or health logs into a behavior mirror. camera into a posture checker or drowsy-driver alert. voice note or dream journal into a mood map or dream interpreter. location into an arrival alarm or neighborhood memory guide. meeting transcript into actions, blockers, and risk summary. text into music, ambience, or a fake trailer. WhatsApp group into a roast generator or inside-joke map. boring PDF into a visual explainer. If they are stuck, ask: what data do you already have, what signal can you capture live, what would be funny enough or useful enough to share in the group today. If they still have no idea: ask what repeated task annoys them every week, what file they never read but should, or what would be funny enough to show this room today. If the idea is too big, say: cut it to one screen, one flow, one output. Force rewrites like: career assistant to resume bullet improver, study platform to PDF to quiz generator, fitness coach to workout log to weekly summary, finance advisor to bank CSV to spending categories.

Five Things That Actually Help

  • 1 Plan before you prompt. Three sentences about what you're building saves 30 minutes of fixing.
  • 2 One thing at a time. One big prompt breaks everything. Three small ones build something.
  • 3 Read what it generates. Code that looks right and code that is right look identical. Until they don't.
  • 4 Test before adding more. The whack-a-mole problem is real. Build 20 lines, test, then continue.
  • 5 Imagination is now the bottleneck. Not your ability to code. Knowing what to build is the hard part.
This is the operating manual for the room. Emphasize 1, 2, and 4: plan before prompting, do one thing at a time, test before adding more. If people start flailing later, come back to these rules. Prompt pattern to give participants: Build a very small web app. User: [who this is for] Input: [the one thing they provide] Output: [the one thing the app returns] Constraints: single page, minimal clean UI, no auth, no database unless absolutely necessary, make the core flow work first. Build the simplest working version, then stop and ask what to improve next.

The Build

Round 1
15 min
Pick the idea. Scope it down.
One user. One input. One output. Start smaller than you want.
Round 2
25 min
Get version one working.
Ugly is fine. Broken edges are fine. Make the core loop work.
Round 3
20 min
Add one thing that feels smart.
Use your data, a live signal, or a better output. Not all three.
Round 4
20 min
Polish, test, publish.
Fix obvious bugs. Add a short intro. Get a link you can share confidently.
Final 20 Min
Demo, debrief, Q&A.
Share links. Talk about what worked, what broke, what surprised you, and what you would do next.
This is the 100-minute build window inside a 2-hour workshop. The remaining time is for opening context, tool setup, and transitions. Keep pushing them to ship by Round 4. In the final 20 minutes, prioritize: one-line demo from each person or team, one lesson learned, one thing they would improve next, then open Q and A. Facilitation by round: Round 1: check that each team has written a 3-sentence spec before prompting. Kill vague ideas early. Round 2: goal is a working version one. If they are styling too early, stop them. Ask: what is the core loop, what is the smallest believable version, what can you delete right now. Round 3: push for one smart moment only. Better prompt quality, real uploaded data, stronger output format, or one useful visualization. Not all of them. Round 4: make them publish, test, add a title, and write a one-line description. Start collecting links before the final shareout. If a build keeps breaking: remove auth, remove database, remove multi-page UI, hardcode sample data if necessary, make the LLM call work first and rebuild outward.

End on the participants, not on yourself. The punchline is that ordinary people in the room shipped something real in one session. That is the proof point. Keep Q and A short if demos are running long. Final debrief angle: AI reduces build friction, but does not remove the need for judgment. Scope discipline is the difference between shipping and stalling. The people who ship fastest usually choose the simplest idea, not the smartest-sounding one. After the workshop: save the best links, note which ideas got traction, write down where people got stuck: ideation, prompting, debugging, publishing.
